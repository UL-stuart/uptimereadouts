## Preprocessing and Normalization

Apply robust text normalization so meaning is preserved across formats.

### Unicode and punctuation

- Normalize to NFKC.
- Convert curly quotes to straight quotes.
- Convert `…` to `....`.
- Collapse multiple whitespace characters into a single space.
- Lowercase for matching, but keep original text for evidence quotes.

### Tokenization and semantics

- Tokenize into words using letters, digits, and `@`.
- Lemmatize where useful, for example `paging` and `paged` should be treated as `page`.
- Use word boundaries when reasoning about single words.
- Prefer semantic equivalence over exact phrases. Treat paraphrases, synonyms, and close variants as matches if the intent is the same.

### Time handling

- Parse timestamps in UTC if available.
- Define `T₀` as the timestamp when Hamed says:  
  `Hey team, we've seen some weird traffic from our data centre network in the last couple of days, it's been mostly minor but never as bad compared to now although.`
- Consider player actions strictly after `T₀` when computing relative timings.
- Round relative timings to whole minutes in `T+X` format.

Ignore this separate phrase unless it is actually part of the transcript and relevant to the analysis:

`Hold on, team. Quick check-in. Is anyone else experiencing internet problems or slow connectivity?`

## Semantic Matching Principles

When judging whether a player meets a criterion, do not rely on exact keywords. Use semantic reasoning.

- Map synonyms and paraphrases to the same intent, for example `loop in security`, `bring SecOps in`, and `page Maya` can all count as security escalation.
- Use windowed co-occurrence where appropriate, for example a directive verb within about 5 to 7 tokens of `Maya`, `SecOps`, or `security team`.
- Respect negation and uncertainty. Do not give credit if the player negates or defers the action.
- Apply actor control. Only give credit if the evaluated player initiates, requests, or leads the action.
- Extract short, direct evidence quotes that show why the criterion is met, partially met, or not met.

## Internal Incident Handling Checklist

For each criterion, evaluate the player’s first qualifying action after `T₀`.

Provide up to 4 evidence snippets from the player’s own messages, ordered earliest to latest. If fewer than 4 exist, include what is available.

### Customer Impact Awareness

- Intent: proactively communicate the outage to customers through a banner, holding page, status notice, or maintenance mode.
- Count as `✅` if the player explicitly requests a customer-facing notice or banner.
- Count as `⚠️` if the player discusses customer communication only tentatively or vaguely.
- Count as `❌` if there is no action or the idea is rejected.

### Customer Impact Assessment

- Intent: quantify impact such as users, orders, regions, severity, failure rate, or revenue at risk.
- Count as `✅` if the player makes a direct and specific request for metrics or scope.
- Count as `⚠️` if the player asks only vague impact questions.
- Count as `❌` if there is no attempt to assess impact.

### Issue Reproduction

- Intent: attempt to reproduce the issue and provide evidence, or direct someone else to do so with evidence requested.
- Count as `✅` if the player reproduces the issue or clearly directs reproduction with evidence.
- Count as `⚠️` if the player only asks whether someone reproduced it, without follow-up.
- Count as `❌` if there is no reproduction attempt or direction.

### Check Platform Changes

- Intent: verify recent deployments, config changes, feature flags, or rollouts.
- Count as `✅` if the player directs or performs a check of recent changes.
- Count as `⚠️` if the player mentions changes but does not initiate a check.
- Count as `❌` if recent changes are ignored.

### Concurrent Investigations

Credit only if led by the player.

- Intent: start and lead multiple investigation streams in parallel.
- Count as `✅` if the player clearly launches 2 or more distinct streams with clear owners and goals.
- Count as `⚠️` if the player makes multiple asks but they overlap or lack clear owners or goals.
- Count as `❌` if the player remains serial or passive.

### Escalation

- Intent: escalate to the correct authority, Hamed or Tinus, without being prompted.
- Count as `✅` if the player initiates direct escalation to Hamed or Tinus.
- Count as `⚠️` if the player only hints at escalation or agrees after someone else proposes it.
- Count as `❌` if there is no escalation.

### Timeboxing

- Intent: assign tasks with explicit time limits and follow-up.
- Count as `✅` if the player sets a concrete timebox and there is visible follow-up or cadence.
- Count as `⚠️` if there is a timebox without follow-up, or only vague urgency like `soon` or `asap`.
- Count as `❌` if there is no timeboxing.

### Leadership and Command

- Intent: provide clear direction, manage threads, maintain urgency, and synthesize status.
- Count as `✅` if there are clear assignments, structure, and periodic summaries.
- Count as `⚠️` if there is only partial direction or unclear ownership.
- Count as `❌` if the player is reactive and does not direct the incident.

### Resolution Verification

Temporal gating is required.

- Intent: confirm restoration only if the issue has actually been resolved, using data and customer confirmation rather than intuition alone.
- Evaluate only if there is evidence the incident reached restoration or closure.
- Prefer evidence from the final phase of the conversation or after explicit restoration signals.
- Count as `✅` if the player provides data-backed confirmation and customer or support corroboration after restoration.
- Count as `⚠️` if verification is premature, anecdotal, or only partial.
- Count as `❌` if there is no verification by the player, or the player claims verification without actual restoration context.

## Timeline and First Occurrence Rule

- Compute all relative times as `T+X` minutes from `T₀`.
- When multiple candidate messages qualify, select the earliest one after `T₀` that sufficiently expresses the intent.
- If the player later gives a stronger version, keep the first qualifying one for timing, but you may mention the later stronger evidence in the rationale if useful.

## Decision Policy and Tie-Breakers

- Prefer explicit directives over weak or hedged statements.
- If a message is ambiguous, use nearby context from the transcript to resolve intent.
- If it remains ambiguous, grade `⚠️` and explain why.
- If the action is performed by others and the player only acknowledges it, do not award credit unless the player clearly initiated or directed it.
